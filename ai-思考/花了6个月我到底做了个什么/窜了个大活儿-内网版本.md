## 1. 一个真实的引子

做云开发平台这几年，我处理过无数次客户工单。

工单的内容大同小异：

> "你们的 DSL 文档我看了，但还是不知道这个字段支持哪些查询操作符？"
> "我照着示例写了，为什么报 `unknown operator`？"

说实话，每次看到这类工单，我心里都有点复杂。

**不是客户的错。**

我确实写了文档，而且自认为写得挺完善了——每个操作符都有说明，每个字段都标注了类型，甚至还配了示例。但现实是：

1. **文档再完善，人也不一定会读** —— 大家都很忙，出问题了才去查，查也不一定查得到
2. **文档是给人读的，但查询是 AI 生成的** —— 客户的 AI Agent 在调我们的 DSL，它读文档的效果，取决于文档质量和模型训练数据
3. **新 DSL 永远面临冷启动问题** —— 刚上线的新查询语法，大模型训练数据里没有，只能靠文档，但文档的覆盖度和可解析性往往不够好

这个问题我琢磨了很久。

直到 AI Coding 兴起，我有了一个不一样的想法：**能不能让 DSL 自己解释自己？**

不需要文档，不需要训练，让 AI 通过协议本身就能知道：这个字段支持哪些查询操作符，那个字段能不能做范围查询。

这就是「自我驱动的文档能力」的起点。

## 2. 什么是模型引擎

先解释一下「模型」是什么。

**模型是数据表的一层抽象**，或者说，是对数据库表带有业务维度的解释。

一张 MySQL 表的 meta 信息是很薄的：`users` 表有 `id`、`name`、`email` 这几个字段，类型分别是 `int`、`varchar`、`varchar`。但下面这些问题，表面数据结构回答不了：

- `email` 是敏感字段吗？前端能直接读吗？
- `age` 字段允许等值查询，还是也允许范围查询？
- 这张表哪些行是"属于当前用户"的？

这些其实是**业务语义**，不是表结构能承载的东西。

所以需要更高维度的抽象——这就是「模型」要解决的问题。

### 四个目标

去年 10 月，AI Coding 刚开始兴起的时候，我给自己定下了四个目标：

1. **安全的 HTTP 能力** —— 把数据库暴露成 HTTP 接口，但必须是受控的、有鉴权的
2. **完善的 RLS 能力** —— Row Level Security，同一张表，不同用户看到不同的行
3. **自我驱动的文档能力** —— AI 不需要人工编写文档，就能自己学会怎么用这个系统
4. **命令行能力** —— 不只是 HTTP API，还能在本地命令行里直接用

抱着这四个目标，我开始了漫长的 Vibe Coding。直到最近，终于把它做出来了。

## 3. AI 自驱是怎么做到的

**如何让 AI 自动理解并正确使用 DSL 的查询语法，而不依赖文档或训练数据。**

### 传统方式的问题

我们设计 DSL 协议的时候，往往离不开查询语法协议的设计。

比如 MongoDB：

```javascript
// 字段 age 是数字类型，支持这些操作符
db.collection.find({ age: { $eq: 25 } })
db.collection.find({ age: { $gt: 18, $lt: 60 } })

// 但字段 name 是字符串，支持的操作符不一样
db.collection.find({ name: { $regex: "^Luke" } })
```

问题在于：**AI 怎么知道 `age` 支持 `$gt`，而 `name` 支持 `$regex`？**

传统做法有两个路径：

**路径一：文档**
把每个字段支持的操作符写进文档。但文档往往覆盖不精细——写到"字段级别"的文档少之又少，而且一定会滞后。

**路径二：模型训练数据**
MongoDB 的 `$eq`、`$gt` 是大模型训练数据里有的知识。但如果你做了一个**新的 DSL**——比如你们公司内部的日志查询语法——大模型没见过，只能靠文档。

这就是我开头说的那个痛点：**做新 DSL 的时候，文档再完善，AI 也不一定读得懂、用得对。**

### 换一条路：让 GraphQL 自解释

我的解法是：**利用 GraphQL 的类型约束，把「字段支持哪些查询操作符」直接写死在 Schema 里。AI 通过 GraphQL Introspection 就能直接知道。**

核心思路是：

1. 不同字段类型，挂不同的 Filter 类型
2. Filter 类型里定义了这个字段**支持哪些操作符**
3. Schema 本身就是结构化元数据，不需要额外文档

举个例子：

```graphql
# String 类型的字段，挂 StringFilter
input StringFilter {
  eq: String
  ne: String
  contains: String
  startsWith: String
  endsWith: String
  in: [String!]
  regex: String
}

# Int 类型的字段，挂 IntFilter
input IntFilter {
  eq: Int
  ne: Int
  gt: Int
  gte: Int
  lt: Int
  lte: Int
  between: [Int!]
}
```

`String` 类型的字段只能挂 `StringFilter`，所以它天然就只支持 `eq`、`contains`、`in`、`regex`；`Int` 类型的字段挂 `IntFilter`，支持 `gt`、`gte`、`lt`、`lte`、`between`。

**这些能力不需要文档说明，写在 Schema 里就是真相。**
其实这个思路和mcp特别像。只不过对于我这种动态字段的接口，mcp无法解决我的问题。

### AI 怎么用

AI 只需要做一次 GraphQL Introspection 查询：

```graphql
query {
  __type(name: "LogFilter") {
    inputFields {
      name
      type {
        name
        inputFields {
          name
          type { name }
        }
      }
    }
  }
}
```

返回结果是完全结构化的 JSON，AI 直接解析就能知道：

- `timestamp` 字段（挂 `DateTimeFilter`）支持 `gt`、`gte`、`lt`、`lte`、`between`
- `level` 字段（挂 `StringFilter`）支持 `eq`、`in`、`regex`
- `message` 字段（挂 `StringFilter`）支持 `contains`、`regex`

然后生成查询时，AI 就不会写出不合法的语法。

用户提问：

> "查询 2024-01-01 之后的 ERROR 日志"

AI 的推理过程：

1. Introspection 告诉我 `timestamp` 字段支持 `gt` 操作符
2. `level` 字段支持 `eq` 操作符
3. 生成合法的 GraphQL 查询：

```graphql
query {
  logs(filter: {
    timestamp: { gt: "2024-01-01T00:00:00Z" }
    level: { eq: "ERROR" }
  }) {
    id
    message
    timestamp
  }
}
```

### 对比：传统方式 vs GraphQL 自解释

| 维度 | 传统方式（文档 / 训练） | GraphQL 自解释 |
|------|------------------------|--------------|
| **元数据来源** | 文档、注释、训练数据 | GraphQL Schema + Introspection |
| **实时性** | 文档可能过时 | Schema 即真相（Single Source of Truth） |
| **精细度** | 通常只到"API 级别" | 字段级别、类型级别 |
| **AI 解析难度** | 需要 RAG / 文档检索 | 直接解析结构化 JSON |
| **扩展性** | 新增字段 → 更新文档 → 重新训练 | 新增字段 → Schema 自动更新 |

**一个实质优势**：不需要 RAG，不需要 Fine-tuning，结构化元数据对 AI 是天然友好的格式。

### 方式一：CLI + AI Agent

这是我最推荐的使用方式。

假设你在 CodeBuddy 里，只需要说一句话：

```
用我本地的 mc 命令，token 是 mc_pat_xxxxxx，帮我查询 orders 表
```
> 这里有时候会有个问题， 就是ai会默认把mc 理解为minio client， 这里实际上是你本地的一个命令

4. 生成正确的 GraphQL 查询
5. 把结果整理成表格返回给你

整个过程里，你不需要懂 MySQL，不需要手写 SQL，甚至不需要知道 GraphQL 的语法。 当然，这是ai带来的便利。 
其实本质就是把你的所有接口设计全部schema化，就能做到这一点。 只不过，在以前，作为人类来说，会觉得graphql复杂。restful更加简单方便。但是对于AI来说，这种“复杂”反而是更好的设计

## 6. 完全体：AI 自驱的系统

整套系统对用户来说是 **AI 自驱的**。

用户通过 GraphQL 自我检索、自动完成使用——GraphQL 的强类型约束天然充当了"护栏"，语法错误可以被 AI 自我感知并纠正，不需要人工介入。

### 管理员负责什么

管理员做的事，本质上是**划定边界**：

1. **数据库 & 表的接入控制**：决定哪些数据库、哪些表对外暴露，不暴露的对任何人都不可见。
2. **RBAC 权限设计**：为不同角色分配不同的读写权限，确保用户只能操作被授权的资源。
3. **RLS（Row Level Security）**：对标 PostgreSQL 的行级安全策略，做到同一张表里，不同用户只能看到属于自己的数据行（该能力正在开发中）。

管理员不需要关心用户怎么查数据，只需要把"能查什么"的边界划清楚，剩下的交给 AI 和 GraphQL 自动完成。

### 普通用户

普通用户只有数据的使用权限。通过 CLI 调用，或者使用开放式API。

## 7. 使用场景

### 我最初设想的场景

最开始我想到的场景很朴素：**我在云上有一个自己的数据库，我希望给它加一层 AI 自驱的 HTTP 能力。**

基于这个出发点，延伸出了几个具体的使用方向：

1. **个人 / 小团队的低代码前端**：前端开发者通过 Vite 等平台快速搭建页面，数据层直接对接 Modelcraft，不需要再写一套后端接口。
2. **测试 & 体验环境的 Harness**：部署一个专门对接测试库或 Demo 库的实例，作为自动化测试流水线的一环，AI 自动生成查询、验证数据，省去手写 fixture 的成本。
3. **云平台售卖数据库能力**：这是我觉得最匹配的场景——云厂商用 Serverless 底层数据库资源，通过 Modelcraft 这一层对个人开发者提供开箱即用的数据库能力，类似 Supabase 的定位，但查询入口是 AI 自驱的 GraphQL，而不是需要自己手写 SQL。

### 一个我至今没想明白的问题

不论哪种场景，有一个绕不开的核心矛盾：

> **Modelcraft 代理了数据库。那么当数据库挂了，算谁的责任？**

如果是用户自己的数据库，Modelcraft 只是一个代理层，数据库宕机理论上与我无关。但用户的感知是"你的服务挂了"。

如果是云平台模式，底层数据库资源由平台提供，责任边界更模糊——是数据库层的问题，还是 Modelcraft 层的问题，用户分不清楚，SLA 也很难定义。

这个问题在 Supabase、PlanetScale 这类产品上同样存在，但他们是自己托管数据库，责任主体是清晰的。Modelcraft 做的是**代理**，不拥有数据，这让责任归属变得很微妙。

我目前没有一个满意的答案，欢迎有想法的同学在评论里和我探讨。
