
在维护使用的过程中，我发现这样一个问题，就是我们的sdk无法完成自解释。虽然它的能力很完善，但是比如说我们目前某个操作doc.where不支持事务， 或者做一个比较复杂的lookup能力时，只能体现在文档里。还有，因为他本质是mongo-like数据库的子集，客户端自己有时候也不确定哪些字段存在，缺少一个标准的schema。 再就是mongo的一些通病，一个字段可以即是字符串也可以是bool。 这些点在AI时代的表现就是有这个能力，但是用起来没那么顺。

我们都知道，在当今时代，一个可以自解释的服务或者工具在AI时代效率会被放大好多倍，即自解释性要远远大于便利性。 所以从产品的角度出发，以前经常有个痛点叫：客户使用门槛高。这个问题我认为是AI时代最适合解决的问题。 mcp 和 cli现在是AI工具的标配，或者说，不提供mcp的情况下， openapi + 文档也是非常AI友好的。 

在牟定这个想法后。我想做的是一个可以自解释的DbaS服务。

### 灵感出现——燃烧小火苗
最开始其实我并没有想到要做一个Saas服务，仅仅是思考一个问题。为什么我们的这套sdk不好用？他在我心目中属于70~80分之间，但是没有达到90分的标准。在我使用过superbase和prisma后。我在思考我们之间的差距到底在哪里？

这个问题一直萦绕在我心头，就像高考数学的最后一道大题一样，即使我不会，我也会花好多太天的时间去重新思考它，然后到理解它为止。 偶然期间，我读了一本书叫做《凤凰架构》。书里介绍了Graphql这样一种协议。 

> 书里引出 GraphQL：它是 Facebook 提出并开源的一种面向资源 API 的数据查询语言，虽然叫“查询语言”，但并不只做查询，也涉及 CRUD。相比“依赖 HTTP、没有额外协议”的 REST，GraphQL 更像一种 带类型系统、带查询语言、带执行规则的 API 协议层。

我自己工作的第一个项目和json schema息息相关。所以我天然对这种schema-based， 类型系统的协议层感情趣。在花了一段时间通读了一遍协议文档后。我的心中燃烧了一个小火苗。


### 无可厚非的前段
最初的想法，大概是做一个类似插件类的proxy后端。负责增强Mysql，对我来说就够了。 但是随着不断开发，不断反思，问题就来了。 

我心目中的想法是做一个graphql作为核心，能够拥有完善的RLS控制的一套Http接口。 同时呢，又不能过度侵入业务。
第一个问题来了：我需要知道一个表的字段以及类型。 我不可能一直实时的查询数据库，同时呢，我又不太想直接作为lazy的模式，即读第一次后加载。我希望这个东西类似schema一样存下来。所以有了设计态的概念。 即此时操作模型的DDL，把Mysql的字段呢进行简化，简化为特定语义的字段。 比如对于使用者来说，其实不太关心varchar，text。只关心他是一个字符串。同理，包括数字（默认浮点数） + 整数的组合也覆盖了数字，然后就是布尔值。 至于复杂对象，则被我归结到关联关系里。

那么做一个web前端自然无可厚非。

### 自然而然的B2B2C架构
我最初是想直接采用Casdoor。但是这个东西太重了。其实在我的系统用户是非常薄的一层。思考再三后，直接vibecoding了。架构这里采取了B2B2C的经典架构。毕竟我自己是做云的人，思维天然被植入了租户，平台这些东西。对我来说，我的客户应该是ISV，是开发者，是管理员。 我要把复杂的模型设计，定制化的权限全都做成灵活的功能，交出去。

### 网关的选型？
我在要不要做网关的事情上纠结了很久。一开始自己AI了一个网关，后来感觉太定制化，最终选择了apisix，完美符合我的要求。

### 复杂就一定门槛高？
虽然大家可能还没了解我做的是一个什么东西，但是听起来就感觉很复杂。复杂就一定门槛高么？不然，在AI时代，agent是解决产品复杂的银弹。当然，前提是你的产品是精确且复杂，而不是臃肿。顺便学习agent，我加入了一个agent服务。 

### 实践就是最好的老师
在实践中，我遇到过很多问题。虽然说，现在AI coding已经降低了门槛，但是很多时候，有很多东西是超出你的认知范围的。 比如我对BFF的认知，最开始我完全不知道什么是BFF，我一直以为的前段就是返回html + css + js代码的一个静态服务器，毕竟所有的运算都是在浏览器上的。后面我会详细介绍。
