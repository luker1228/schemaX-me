所谓部署，究竟在部署什么？

你真正要部署的，不是源码，而是编译完以后那个“可运行的东西”。

代码写完、编译之后，通常会产出一个可运行的产物 artifact：Go 会产出二进制文件 myapp，Java 会产出 JAR 包 app.jar，前端会产出一组静态文件 dist/。

这些产物本身就可以被运行。Go 二进制能直接启动，JAR 交给 JVM 就能起进程，前端静态文件交给 Web 服务器就能对外提供页面。平时本地联调，本质上就是在跑这个产物。

打包成镜像，本质上就是把产物连同运行环境封进一个标准盒子里：源码 --编译--> 产物 artifact --打包--> 镜像 image。

DOCKER-02 问题

镜像里究竟放什么？

A: 本地编译好,直接 COPY 进镜像

FROM alpine:3.19
COPY myapp .
CMD ["./myapp"]

简单，但有个致命陷阱:本地构建环境和镜像运行环境不一致，产物可能根本跑不起来。

  • 编译器版本不一致：本地用 Go 1.21 编译，运行环境却按 Go 1.16 的预期去跑，行为可能异常甚至直接失败
  • 操作系统不一致：你在 macOS 上编出来的是 darwin 二进制，丢进 Linux 镜像就会报 exec format error
  • C 库不一致：在普通 Linux(glibc)上编译，再扔进 alpine(musl)，就可能缺动态库

B: 在镜像里编译

FROM golang:1.22
WORKDIR /app
COPY . .
RUN go build -o myapp .
CMD ["./myapp"]

环境一致的问题解决了，但新的麻烦马上出现:

  • 镜像巨大：带着整个 Go 工具链和全部源码，生产运行时根本用不到
  • 不安全：镜像里多了编译器、shell、各种工具，攻击面平白无故变大
  • 构建慢：每次都要拉大镜像、装工具链，CI 和发布都会被拖慢

做法 A 死在环境不一致，做法 B 死在臃肿不安全。有没有一种方式，既保证环境一致，又能让最终镜像小而安全?答案就是多阶段构建。

DOCKER-03 解法

把编译产物和运行分开
多阶段构建

核心思想非常简单:一个阶段专门编译，另一个阶段只放产物。最终镜像只保留最后一个阶段，前面的 builder 用完即弃

阶段 1 · builder Go 工具链 + 源码 go build 产物：myapp 用完即弃，只取产物
查看源码 
FROM golang:1.22 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o myapp .
COPY --from=builder
阶段 2 · runtime alpine + myapp 最终部署的是这个镜像
查看源码 
FROM alpine:3.19
WORKDIR /app
COPY --from=builder /app/myapp .
CMD ["./myapp"]

跨阶段复制

COPY --from=builder 是关键：跨阶段只取产物，不把构建工具链带进最终镜像。

朴素(单阶段):  golang + 源码 + 工具链 + 二进制   ≈ 800 MB
多阶段(alpine): alpine + 二进制                  ≈ 20 MB

为什么要追求小？

就三个字：快、省、安全。

  • 快：镜像小，拉取 / 推送快，Pod 启动快，扩容和滚动更新也都更快
  • 省：仓库存储和跨地域传输都按体积计成本
  • 安全：包越少，攻击面越小，CVE 也越少；很多最小镜像连 shell 都没有

一个 Dockerfile，一条 docker build 命令，就可以同时完成“受控编译”和“最小运行”这两件事。

DOCKER-04 运行时

不同语言的产物对应不同运行阶段

核心套路都一样:编译阶段 + 运行阶段。真正变化的，只是运行阶段到底要放什么。

产物类型 运行时依赖 最终阶段
Go / Rust 静态二进制 无 scratch 或 alpine
C/C++ 动态二进制 libc 等动态库 alpine / debian-slim
Java JAR JVM eclipse-temurin:21-jre
前端静态包 Web 服务器 nginx:alpine

Java 例子: JAR → JRE

FROM maven:3.9 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

FROM eclipse-temurin:21-jre-alpine
COPY --from=builder /app/target/*.jar /app.jar
CMD ["java", "-jar", "/app.jar"]

Java 的特殊之处在于:它虽然是跨平台字节码，但运行时必须带 JVM，所以镜像天然比 Go 大。这不是 Dockerfile 写得不好，而是语言运行时本身的成本。

  • JRE 不等于 JDK：生产只要 JRE，别把编译器一起带进去
  • alpine 版更小：eclipse-temurin:*-jre-alpine 通常比 Debian 版小很多
  • fat JAR 不利于缓存：Spring Boot 一改代码，整个 JAR 层都变

Java 镜像想继续瘦，还可以往后做分层 JAR、jlink，甚至 GraalVM Native Image，但那已经是下一层优化了。

前端例子: dist → nginx

FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html

规律很稳定:运行阶段只装“跑产物必需的最小运行时”。Java 只装 JRE，前端只装 nginx，Go 干脆什么都不装。

DOCKER-05 默认习惯

打包好镜像的几个默认动作

  • .dockerignore:把 node_modules、.git、测试数据挡在构建上下文之外
  • 小基础镜像:优先 alpine、-slim、distroless
  • 依赖先拷,源码后拷:利用层缓存，依赖没变就跳过重装
  • 非 root 用户运行:用 USER appuser 降低风险
  • 合并 RUN:多条命令用 && 串起来，减少镜像层数

这里最值得记住的是“依赖先拷、源码后拷”。源码天天变，依赖很少变。顺序写对了，改一行业务代码就不会触发“重装所有依赖”那一层，CI 时间差很多。

DOCKER-06 配置

镜像做好了以后 运行配置怎么办

配置不要打进镜像。镜像应该 immutable，环境差异来自外部注入，而不是打出三份不同镜像。

镜像只装“程序”，但程序启动往往还依赖运行时配置，比如数据库连接、业务参数、密钥文件。

  • 多环境：同一镜像要跑 dev / test / prod，配置天然不同
  • 安全：密码打进镜像，等于把钥匙焊死在门上
  • 解耦：改配置不该重新打镜像，镜像应该是 immutable 的

最常用的做法是:运行时从外部注入配置，尤其是挂载配置文件。

docker run -e CONFIG_FILE=/app/config.yaml \
  -v ./config.yaml:/app/config.yaml \
  myapp:1.0

这里环境变量只负责告诉程序“配置文件在哪”，真正的配置内容是通过 volume 挂进去的。这种方式最大的好处是:你能进容器直接看文件，容易核对。

docker-compose 挂载

services:
  app:
    image: myapp:1.0
    volumes:
      - ./config.yaml:/app/config.yaml
    environment:
      CONFIG_FILE: /app/config.yaml
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: s3cret

同一镜像，不同环境只换挂进去的配置文件。镜像不变，变的是外部配置，这才是正确的部署边界。

少量简单配置可以直接用环境变量，比如 LOG_LEVEL=debug。但配置一多，一复杂，就应当回到文件挂载。

敏感信息

  • 绝不进镜像，也不要明文提交 Git
  • .env 这类文件要进 .gitignore
  • 更复杂的密码管理、动态配置更新，可以留到 K8s / Secret / 配置中心那一层解决
DOCKER-07 总结

一句话总结

打包镜像的本质 = 只把运行需要的放进镜像,其余(编译器、源码、构建工具)全部留在构建阶段。

产物 + 运行时  ──COPY --from──>  最终镜像(小而安全)
(构建阶段产出)                    (只有这一个会被推送/部署)

多阶段构建就是把这件事写成标准流程:前段编译，后段运行，最终镜像又小又安全。这才是从“编译产物”到“生产级镜像”的标准姿势。
