// DEP-01 · DOCKERFILE

从编译产物到镜像

把二进制、JAR 或前端包封进一个可部署的标准盒子。核心动作:多阶段构建。

0. 打包镜像,到底在打包什么?

代码写完、编译之后,通常会产出一个可运行的产物:Go 是二进制文件,Java 是 JAR 包,前端是 dist/ 静态文件。

打包成镜像,本质上就是把这个产物连同运行环境封进一个标准盒子里。

源码 --编译--> 产物 artifact --打包--> 镜像 image

关键认知:镜像里真正运行需要的,只是 产物 + 运行时环境。源码、编译器、构建工具,运行时都不该混进去。

1. 本地构建产物为什么不好?

做法 A:本地编译好,直接 COPY 进镜像

FROM alpine:3.19
COPY myapp .
CMD ["./myapp"]

简单,但有致命陷阱:本地构建环境和镜像运行环境不一致。macOS 编出的 darwin 二进制丢进 Linux 镜像,就会遇到 exec format error。

做法 B:在镜像里编译

FROM golang:1.22
WORKDIR /app
COPY . .
RUN go build -o myapp .
CMD ["./myapp"]

环境一致了,但镜像会巨大、不安全、构建慢。生产运行根本不需要 Go 工具链和源码。

2. 解法:编译和运行分开

多阶段构建用一个阶段专门编译,另一个阶段只放产物。最终镜像只保留最后一个阶段,前面的 builder 用完即弃。

FROM golang:1.22 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o myapp .

FROM alpine:3.19
WORKDIR /app
COPY --from=builder /app/myapp .
CMD ["./myapp"]

COPY --from=builder 是关键:跨阶段只取产物,不把构建工具链带进最终镜像。

3. 不同语言的产物,不同运行阶段

产物类型 运行时依赖 最终阶段
Go / Rust 静态二进制 无 scratch 或 alpine
C/C++ 动态二进制 libc 等动态库 alpine / debian-slim
Java JAR JVM eclipse-temurin:21-jre
前端静态包 Web 服务器 nginx:alpine 
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html

4. 打包好镜像的几个习惯

  • .dockerignore:把 node_modules、.git、测试数据挡在构建上下文之外。
  • 小基础镜像:优先 alpine、-slim、distroless。
  • 依赖先拷,源码后拷:利用层缓存,依赖没变就不用重装。
  • 非 root 用户运行:用 USER appuser 降低风险。

5. 运行配置怎么办?

配置不要打进镜像。同一镜像要跑 dev、test、prod,配置应该从外部注入。

docker run -e CONFIG_FILE=/app/config.yaml \
  -v ./config.yaml:/app/config.yaml \
  myapp:1.0

复杂配置用文件挂载,能查看、能核对。简单配置用环境变量,比如 LOG_LEVEL=debug。

6. 一句话总结

打包镜像的本质 = 只把运行需要的放进镜像,其余全部留在构建阶段。
