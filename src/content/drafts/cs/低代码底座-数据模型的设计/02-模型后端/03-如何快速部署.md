# 从改表到联调：模型后端的快速交付流程

## 背景
AI 写代码的速度我想大家都感受到了。 但是这里作为程序员不仅仅是写代码。你还要完成DB变更，前端联调，服务部署，集成测试等功能。这里有关测试的事情我准备单独去讲

本期先展开DB变更，前端联调，服务部署

## 1. 如何把数据库变更做快做稳

### 问题

数据库部署的问题在哪里呢？

AI Coding的时代。首先AI要有一份自己对数据库的认知。它得知道你维护的表结构长什么样子。

> 其实我认为在AI时代，这个数据库的Schema是你一定要review的东西。就是你不能AI半天，你连你自己核心的表长什么样子都不知道。

在最开始的编程中，AI其实做的非常好。但是迭代几轮之后，ALTER 字段几轮之后。AI就对这个信息不全了。如果我们用的是类似Gorm这样的框架，我们可以通过go的orm tag字段来维护。但是有个问题，就是会丢失索引和varchar， text等信息。即如果我的代码换到另一个环境去部署。我不知道加啥索引。有些字段设置varchar32，64还是多少合适。

那么此时来说，对于我们维护一个create table语句就是最适合的事情。

但是呢，在开发测试阶段，有时候我们需要删除几个字段。  我们就只能继续追加 ALTER语句。时间长了，其实我们相当于维护了一套Sql的变更。

这件事情本身没有什么问题。只是数据库的变更这种东西，他试错的成本太高，有没有revert。所以我们自己维护其实很费心智，哪怕是AI维护，我也没有找到最佳的实践方法，还是不放心。

所以我就找了很多工具，测试了很多工具。最终找到了altas。

具体 Atlas 的基础命令我就不展开了。这里我其实推荐保留两套用法：

1. 测试环境快速验证，直接用 `schema apply`
2. 正式环境和团队协作，走完整的 migration 流程

### 1.1 测试环境的快速方法

测试或者开发环境快速使用的方法

```bash
atlas schema apply \
  -u "${DB_URL}" \
  --to file://db/schema/mysql/ \
  --dev-url "${DB_DEV_URL}" \
  --auto-approve
```

| 参数 | 含义 |
| --- | --- |
| `-u ${DB_URL}` | 目标库（current state，Atlas 连上去 inspect 当前 schema） |
| `--to file://db/schema/mysql/` | 期望态（你的 SQL/HCL schema 目录） |
| `--dev-url ${DB_DEV_URL}` | dev 临时库（Atlas 在里面建临时 schema 做 normalize + 模拟 plan，不会改你真实库） |
| `--auto-approve` | 直接执行变更，适合测试环境快速验证，不适合直接拿来跑生产 |

这套方式的优点是快。你改完 schema 之后，马上就能把测试库推到最新状态，特别适合本地开发、自测、临时联调环境。

但它更适合“快速验证”，不适合承担正式发布流程。因为真正到了线上，你还是需要一份可审计、可回放、可追踪的 migration 历史。

### 1.2 一套完整的线上流程：从 0 接入到日常迭代

这一套流程的核心思想很简单：

1. `db/schema/mysql/` 维护期望态
2. `db/migrations/` 维护真实执行过的变更历史
3. 开发时改 schema，发布前由 Atlas 生成 migration
4. 测试库和生产库都只认同一套 migration

#### 阶段一：项目初始化

先把目录固定下来：

```bash
mkdir -p db/{schema/mysql,migrations}
```

然后把线上当前 schema dump 成你的起始期望态：

```bash
atlas schema inspect \
  --url "${PROD_DB_URL}" \
  --format "{{ sql . }}" \
  > db/schema/mysql/_base.sql
```

这一步完成后，`db/schema/mysql/_base.sql` 就是“线上此刻的完整全貌”。后面你维护的重点，不再是直接改线上库，而是持续修改这个 schema 目录。

#### 阶段二：老库首次接入 Atlas

如果你的库不是从 0 开始，而是已经在线上跑了一段时间，那么第一次接 Atlas 时，需要先做 baseline。

先根据 schema 目录生成 baseline migration：

```bash
atlas migrate diff baseline \
  --dir file://db/migrations \
  --to file://db/schema/mysql \
  --dev-url "${DB_DEV_URL}"
```

执行后会生成类似下面这些文件：

- `db/migrations/20260102xxxxxx_baseline.sql`
- `db/migrations/atlas.sum`

接着在生产库上做“已执行标记”，而不是重新执行 baseline SQL：

```bash
atlas migrate apply \
  --url "${PROD_DB_URL}" \
  --dir file://db/migrations \
  --baseline 20260102xxxxxx
```

这里最关键的一点是：生产库第一次接入时，`--baseline` 必须传。否则 Atlas 会真的去执行 `baseline.sql`，而线上本来就已经有这些表了，通常会直接报 `table already exists`。

如果是开发库或者测试库第一次接入，就可以直接执行：

```bash
atlas migrate apply \
  --url "${DB_URL}" \
  --dir file://db/migrations
```

因为这些环境本来就是空库，baseline 本来就应该真实建表。

#### 阶段三：日常开发迭代

后续日常开发时，不是直接手写一堆零散 ALTER，而是先修改期望态 schema。

比如你要给 `users` 表新增一个 `phone` 字段，先改：

```bash
vim db/schema/mysql/users.sql
```

改完之后，再由 Atlas 生成增量 migration：

```bash
atlas migrate diff add_user_phone \
  --dir file://db/migrations \
  --to file://db/schema/mysql \
  --dev-url "${DB_DEV_URL}"
```

这时会生成类似：

```text
db/migrations/20260103xxxxxx_add_user_phone.sql
```

然后把 schema 变更和 migration 一起提交到 PR。

这里有两个习惯我觉得很重要：

1. `--to` 始终指向 `db/schema/mysql`
2. 不要手工去改已经生成的历史 migration，除非你明确知道自己在处理什么问题

因为一旦 schema 目录和 migration 历史开始各写各的，后面就很容易漂移。

#### 阶段四：上线生产

生产发布前，我建议至少走三步。

第一步，先看当前有哪些 pending migration：

```bash
atlas migrate status \
  --url "${PROD_DB_URL}" \
  --dir file://db/migrations
```

第二步，先做 dry-run，看 Atlas 计划执行什么：

```bash
atlas migrate apply \
  --url "${PROD_DB_URL}" \
  --dir file://db/migrations \
  --dry-run
```

第三步，再正式执行：

```bash
atlas migrate apply \
  --url "${PROD_DB_URL}" \
  --dir file://db/migrations \
  --tx-mode none
```

这里 `--tx-mode none` 主要是因为 MySQL 下很多 DDL 本身就不支持事务，强行按事务包起来没有意义。假如你的目标库是 Postgres，再去考虑是否需要事务回滚能力。

如果是多人协作或者 CI 自动发版，我还建议额外加锁，避免两个发布流程同时改库：

```bash
atlas migrate apply \
  --url "${PROD_DB_URL}" \
  --dir file://db/migrations \
  --tx-mode none \
  --lock-timeout 30s
```

#### Atlas 稳定交付机制

- 新项目能快速接入
- 老库能平滑纳管
- 日常迭代只改 schema，不直接碰线上
- 发布前先看 pending，再 dry-run，再 apply
- 测试库和生产库共用同一套 migration 历史

当这套节奏固定下来之后，数据库变更这件事就不再依赖“某个人特别小心”，而是依赖流程本身足够稳定。

## 2. 屠龙术——真正稳定AI部署的方法

很多人用AI部署的时候，会遇到一个问题。就是发现AI每次部署的时候，看起来当时成功了，后面下次用的时候AI部署又遇到各种问题， 又失败了。循环往复这个过程。 

这里其实有2个核心问题。

1. 你不懂这个部署流程。 说白了，你相当于是暴力让AI帮你解决的。

> 我这个有点暴论。可能我的知识还是有点老，在我看来，单论部署这个问题，我认为部署是个需要知识量广度 + 体力的事情。所以AI可以帮你选择方案，可以帮你完成体力。但是固化的事情还是需要个人的知识量。

2. 我推荐固化部署的方式就是使用makefile or justfile

这里需要的就是一个快速支持命令使用的方式。 这里makefile 或者 justfile。

我选择justfile，纯粹是justfile语法简单，但是问题就是justfile一般的linux系统不会默认带。所以需要安装。justfile语法比较简单，写完我可以大概review下就明白AI想干嘛。

> 当我用一个新技术的时候。我其实不会一上来就让AI用，而是让AI教我，最起码让我能大概明白AI在干什么的时候，才会开始这个飞轮。因为我是要持续交付的，不是写个demo给大家，就完事了。



## 3. 如何和前端联调

如果你认同我的 code-gen 思想。这里和前端联调其实你什么都不用干，只需要把这里的 GraphQL SDL 给前端，然后给出前端你暴露的 GraphQL endpoint，这个事情就解决了。

这里我想讲的更加深入一部分。这个时代，像我所在的小组已经开始一人前后负责，即不再区分前后端，需求都是一人处理了。所以这里我认为所有后端要开始逐步学习前段。

我这里前端依然采取 codegen 的思想，而且做得更完整一些。

以 `modelcraft-front` 这个项目为例，它前端 GraphQL codegen 这块，实际用的是下面这一套技术组合：

- `@graphql-codegen/cli`
- `@graphql-codegen/typescript`
- `@graphql-codegen/typescript-operations`
- `@graphql-codegen/typescript-msw`
- `@apollo/client`

它的运行方式也很直接。

第一，schema 输入直接读取后端交付过来的 SDL 文件：

```text
contract/graph/org/schema/*.graphql
contract/graph/project/schema/*.graphql
```

第二，前端把自己真正会调用的 query 和 mutation 写在 `src/api-client/**/*.ts` 里，形式就是 Apollo 的 `gql` 文档。

第三，执行下面这个命令：

```bash
npm run codegen
```

对应脚本其实就是：

```bash
graphql-codegen --config codegen.ts
```

然后它会生成三类东西。

第一类，是前端真正消费的 TypeScript 类型：

```text
src/generated/graphql.ts
```

这里面会生成 operation 对应的入参、出参、枚举、标量映射。比如 `DateTime -> string`，`ID -> string`。这样前端写页面时，拿到的就是明确的类型约束。

第二类，是基于 GraphQL operation 生成的 MSW mock handlers：

```text
src/mocks/handlers/org/generated.ts
src/mocks/handlers/project/generated.ts
```

这点我觉得很重要。因为这意味着 codegen 不只是服务于“联调期”，它还顺手把 mock 能力也补齐了。也就是说，当后端还没完全 ready 的时候，前端其实已经可以拿着同一份 schema 和 operation 做本地开发、页面验证、测试回归。

第三类，是运行时请求本身依然走 Apollo Client。也就是说：

1. `gql` 文档手写
2. 类型由 codegen 生成
3. 请求由 Apollo Client 发出
4. endpoint 通过项目上下文动态拼接

在 `modelcraft-front` 里，project 级别的 GraphQL 地址大概就是这种形式：

```text
/api/bff/graphql/org/${orgName}/project/${projectSlug}
```

如果是 runtime 级别，还会继续细分到：

```text
/api/bff/graphql/org/${orgName}/project/${projectSlug}/db/${databaseName}/model/${modelName}
```

所以你会发现，这种联调模式下，后端真正需要交付给前端的东西并不多：

1. 一份稳定的 GraphQL SDL
2. 一个清晰的 GraphQL endpoint 约定
3. 少量必要的上下文规则，比如 org、project、runtime 这些 scope 怎么切

剩下的类型生成、请求拼接、mock、页面接入，前端都可以自己往前推进。

### 3.1 这套联调方式为什么省事

传统 REST 联调经常卡在一个地方：前后端各自维护一套接口认知，而且这两套认知很容易漂移。

但 GraphQL + codegen 的方式，本质上是在把“接口契约”前置成一份由后端交付、前后端共同依赖的源文件。你一旦把 SDL 定稳，前端就能立刻做几件事：

- 生成 TS 类型
- 写 query 和 mutation
- 跑 mock
- 直接接页面

也就是说，联调可以变成“前端先按契约推进，后端再把 resolver 和数据补齐”。

这对一人前后端负责尤其有价值。因为你自己切换上下文的时候，不需要脑子里维护两份协议。后端交付的 SDL 就是这一轮联调里的唯一真相源。

### 3.2 后端要学前端，具体先学什么

我这里的观点很明确：以后端为主的人，不一定要马上把前端学到很深，但至少要先学会下面几件事：

1. 能读懂前端的 GraphQL query 和 mutation
2. 知道 Apollo Client 这种客户端是怎么发请求的
3. 知道 codegen 是怎么从 SDL 生成类型的
4. 知道 mock handler 是怎么让页面先跑起来的

你一旦理解了这四件事，所谓“联调”这个词，很多时候就会被你拆解成一套很普通的工程流程，而不再是一个高摩擦的协作环节。

### 3.3 codegen 思想真正解决的是什么

我一直强调 codegen，原因很简单，它在 AI 时代特别适合持续交付。

原因很简单：

1. SDL 是结构化输入，AI 很容易理解
2. 生成代码是稳定体力活，适合交给工具
3. 类型错误会尽早暴露，不会拖到联调后期
4. mock、前端接入、后端实现都能围绕同一份契约展开

所以如果你问我，前后端联调到底怎么提速，我的答案就是尽量把接口协作变成一套可生成、可校验、可复用的工程体系。
