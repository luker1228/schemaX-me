# 从改表到联调：模型后端的快速交付流程

## 背景
AI 写代码的速度我想大家都感受到了。 但是这里作为程序员不仅仅是写代码。你还要完成DB变更，前端联调，服务部署，集成测试等功能。这里有关测试的事情我准备单独去讲

本期先展开DB变更，前端联调，服务部署

## 1. 如何把数据库变更做快，又做稳

### 问题

数据库部署的问题在哪里呢？

AI Coding的时代。首先AI要有一份自己对数据库的认知。它得知道你维护的表结构长什么样子。

> 其实我认为在AI时代，这个数据库的Schema是你一定要review的东西。就是你不能AI半天，你连你自己核心的表长什么样子都不知道。

在最开始的编程中，AI其实做的非常好。但是迭代几轮之后，ALTER 字段几轮之后。AI就对这个信息不全了。如果我们用的是类似Gorm这样的框架，我们可以通过go的orm tag字段来维护。但是有个问题，就是会丢失索引和varchar， text等信息。即如果我的代码换到另一个环境去部署。我不知道加啥索引。有些字段设置varchar32，64还是多少合适。

那么此时来说，对于我们维护一个create table语句就是最适合的事情。

但是呢，在开发测试阶段，有时候我们需要删除几个字段。  我们就只能继续追加 ALTER语句。时间长了，其实我们相当于维护了一套Sql的变更。

这件事情本身没有什么问题。只是数据库的变更这种东西，他试错的成本太高，有没有revert。所以我们自己维护其实很费心智，哪怕是AI维护，我也没有找到最佳的实践方法，还是不放心。

所以我就找了很多工具，测试了很多工具。最终找到了altas。


### 1.2 如何快速部署测试数据库
测试数据库的目标，不是“做一个完美的生产镜像”，而是“尽快提供一个可以验证变更的环境”。

这里我更推荐用下面这个思路：

1. 先固定一套建库基线
2. 再固定一套初始化数据
3. 最后把整个流程脚本化

也就是说，你需要的不是一份“怎么手工建库”的文档，而是一套可以重复执行的机制：

```text
创建数据库
-> 执行基线表结构
-> 执行所有增量 migration
-> 导入最小可验证测试数据
-> 应用启动并连通验证
```

这里有几个经验非常重要。

第一，不要让测试库依赖手工维护。只要测试库还需要人工点来点去、复制 SQL、临时改字段，它就不可能稳定。

第二，测试数据不要追求“大而全”，而是要追求“能覆盖关键链路”。比如：

- 一个最小租户
- 一组典型模型
- 几条带关系的数据
- 几条边界数据，比如空值、重复值、非法状态

第三，建库流程一定要可重复执行。因为后面你会频繁遇到：

- 新同学本地起环境
- 新测试环境拉起
- 数据污染后重新初始化
- 某个分支需要验证一组新 migration

如果没有自动化脚本，后面每个人都会用自己的方式起库，环境一致性很快就失控。

### 1.3 如何稳定修改正式数据库
测试库强调快，正式库强调稳。这两者不能混为一谈。

正式数据库变更最忌讳的，是把“改完结构”和“切换代码逻辑”放在同一个瞬间完成。因为一旦失败，回滚会很痛苦。

我更推荐的方式是把正式库变更拆成几个阶段：

1. 先做兼容性变更
2. 再发布兼容这次变更的服务版本
3. 等流量和数据稳定后再做收尾清理

比如一个常见案例：把旧字段 `status` 拆成 `biz_status` 和 `sys_status`。如果你直接删旧字段，再让新代码上线，风险会很高。更稳的方式通常是：

1. 先新增 `biz_status`、`sys_status`
2. 写迁移脚本，把旧数据回填过去
3. 服务代码同时兼容新旧字段
4. 验证稳定后，再停止读取旧字段
5. 最后再删除旧字段

这套做法的核心思想只有一句话：

**数据库结构的演进要先于业务逻辑的硬切换。**

只要你接受这件事，很多线上改库的风险都会显著下降。

### 1.4 如何保证测试库和正式库的一致性
很多团队会说，我们测试环境和正式环境不一致，所以测试结论不可信。

但这里要先说清楚，一致性不代表“数据完全一样”。真正重要的是下面两层一致：

1. 表结构一致
2. 变更路径一致

也就是说，测试环境和正式环境都应该通过同一套 migration 演进，而不是一个靠脚本，一个靠手改。

我自己更看重下面这几个原则：

- 所有结构变更必须进入版本管理
- 测试环境先执行 migration，正式环境后执行 migration
- 不允许绕开 migration 手工改正式库
- 每次发版都能明确对应到一组数据库变更

这里最危险的事情，其实不是“脚本写错了”，而是“有人在线上偷偷补了一下”。因为一旦开始允许手工改库，环境漂移就会迅速累积。到最后，测试通过不代表线上能过，线上出问题也无法在测试环境复现。

所以保证一致性的关键，不是大家更仔细，而是流程上根本不给“随手改一下”留下空间。

### 1.5 数据库变更真正应该沉淀成什么
最后你要沉淀下来的，不是一堆零散 SQL，而是一套稳定的数据库交付机制：

- 新环境可以快速起库
- 结构变更可以重复执行
- 测试先验证，正式再发布
- 改库过程有兼容期，不靠一次性硬切

当数据库变更进入这种节奏后，后端交付速度会明显上升，因为你不会再把大量时间浪费在“这张表现在到底长什么样”这种问题上。

## 2. 如何快速和前端联调

### 2.1 为什么前后端联调总是慢
联调慢，很多时候不是技术问题，而是交付顺序有问题。

常见情况是这样的：

1. 后端先改表
2. 后端再改接口
3. 后端自己本地测一下
4. 等有空了再告诉前端可以联调

这套流程最大的问题是，前端拿到环境的时候，很多不确定性都还没有消失：

- 字段含义有没有完全定
- 接口返回结构会不会再改
- 测试数据够不够支撑页面开发
- 服务部署之后是不是马上又要回滚

所以前端并不是在和一个稳定系统联调，而是在和一个还在抖动的目标联调。

### 2.2 联调前应该先对齐什么
我觉得联调前至少要先对齐 3 层东西。

第一层是模型层。也就是字段本身的定义：

- 字段是什么含义
- 是否必填
- 默认值是什么
- 枚举范围是什么
- 是否允许为空

第二层是接口层。也就是前端真正消费的数据结构：

- 入参怎么传
- 出参长什么样
- 分页、排序、筛选怎么表达
- 错误码和错误提示如何约定

第三层是场景层。也就是页面实际会怎么使用：

- 列表页看哪些字段
- 详情页需要哪些关联数据
- 编辑页提交时哪些字段可改，哪些字段只读
- 边界状态下页面应该如何表现

如果这三层没有提前对齐，联调时就会频繁出现一种情况：接口 technically 能通，但是业务上不能用。

### 2.3 如何把联调提前
高效联调的关键，不是“后端写快一点”，而是尽量把联调从“最后一步”前移。

这里我比较推荐下面这个顺序：

1. 先把模型字段和接口形状定下来
2. 后端尽快给出可访问的测试接口
3. 准备一批稳定的测试数据
4. 前端提前按约定结构接入页面

注意，这里最重要的不是接口已经 100% 完成，而是接口的外部形状已经稳定。比如：

- 字段名字尽量别再改
- 列表返回结构尽量别再改
- 错误格式尽量别再改

只要这几层外部契约先稳定，前端其实可以很早开始干活，不需要等你把所有内部逻辑都完全打磨完。

从这个角度看，联调前移的本质，是尽快让前端依赖“稳定契约”，而不是依赖“完整实现”。

### 2.4 如何减少联调中的反复修改
联调最浪费时间的，不是 bug，而是来回推翻。

比如前端刚接完一个字段，你又说字段要改名；刚调完列表页，你又说分页协议要变；刚写完表单校验，你又说某个字段其实允许空值。

这些改动在后端看来也许都不大，但对前端来说，每一次都可能牵动：

- 类型定义
- 状态管理
- 表单逻辑
- 页面展示
- 联调数据

所以后端在联调阶段要尽量控制一个原则：

**内部实现可以迭代，外部接口尽量稳定。**

如果确实要变，也尽量优先做兼容，而不是直接打断前端现有逻辑。

另外，联调时不要只盯着某一个接口通不通，更要验证完整业务链路。例如：

```text
创建模型
-> 发布模型
-> 前端查询列表
-> 前端打开详情
-> 前端编辑并提交
-> 后端再次查询验证结果
```

很多问题单看单个接口发现不了，只有放到完整链路里才能暴露出来。

### 2.5 快速联调真正依赖什么
最后总结一下，前后端联调要快，依赖的通常不是某种神奇工具，而是下面 3 件事：

1. 接口契约尽早稳定
2. 测试环境尽早可访问
3. 测试数据尽早可复用

也就是说，联调速度本质上是环境准备能力和接口稳定能力的体现，而不是谁加班更晚。

## 3. 如何快速部署服务

### 3.1 服务部署为什么也会成为瓶颈
很多团队把数据库、接口、服务部署分开看，所以每个环节都在单独优化。但在真实交付里，服务部署慢会直接反向拖慢前面所有动作。

比如下面这种情况就非常常见：

- 数据库已经改好了，但是服务还没发
- 服务已经发了，但是环境变量没配齐
- 测试环境已经起了，但是前端访问不到
- 前端准备联调了，结果发现部署的是旧版本

这说明问题不在“会不会部署”，而在“部署是否已经标准化”。

### 3.2 快速部署的前提是什么
我认为一个服务想要部署得快，至少要满足 4 个前提：

1. 构建流程标准化
2. 配置项清晰且可管理
3. 服务版本和数据库变更有明确关系
4. 测试环境可以频繁、低成本地更新

如果这 4 点做不到，每次发版都会带着很多隐性不确定性。你以为是在发服务，其实是在现场排查环境问题。

### 3.3 如何把部署收敛成标准动作
一个成熟的快速部署流程，最好能收敛成固定动作，而不是依赖某一个熟练同学手工操作。

理想情况下，一次交付应该尽量接近下面这条链路：

```text
代码合并
-> 自动构建产物
-> 部署测试环境
-> 执行数据库 migration
-> 回归关键链路
-> 通知前端开始联调
-> 确认后再发布正式环境
```

这里有一个常见误区，就是把“部署成功”理解成“服务进程起来了”。

其实对于模型后端来说，真正的部署成功至少要包含：

- 服务进程正常
- 配置加载正确
- 数据库连接正常
- 当前版本接口可访问
- 关键业务链路可执行

如果这些没有一起验证，那只能叫“代码传上去了”，不能叫“可交付了”。

### 3.4 服务部署和数据库变更怎么配合
服务部署和数据库变更一定要一起设计，不然两边会互相制造风险。

这里比较稳妥的一种模式是：

1. 先发布兼容数据库旧结构和新结构的服务
2. 再执行数据库变更
3. 验证新逻辑稳定后，逐步去掉旧兼容代码

这样做的好处是，服务和数据库不会在同一个瞬间同时发生“硬切换”。

因为一旦你做成下面这种顺序：

```text
先强改数据库
-> 再上线依赖新结构的服务
```

那中间任意一个环节失败，系统就可能进入不可用状态。

所以更稳的思路始终是：

**先发布兼容版本，再逐步完成结构迁移。**

这个原则不仅适合数据库字段变更，也适合索引变更、关系拆分、协议演进。

### 3.5 快速部署真正解决的是什么
快速部署不只是为了省几分钟，它真正解决的是交付链路里的等待时间和不确定性。

当服务部署足够标准化以后，数据库变更、接口联调、问题修复都会明显加快。因为大家不用再把时间浪费在：

- 环境到底有没有更新
- 这个版本到底发没发上去
- 前端看到的是不是最新逻辑
- 数据库脚本和代码版本是不是匹配

部署一旦稳定，整个团队的交付节奏都会更顺。

## 总结
模型后端的快速交付，本质上不是把某一个动作做到极致，而是把整条链路串起来。

第一，是数据库变更要形成标准机制。测试库起得快，正式库改得稳，测试和正式通过同一套 migration 保持一致。

第二，是前后端联调要尽量前移。尽早稳定接口契约，尽早提供测试环境，尽早准备联调数据。

第三，是服务部署要足够标准化。让数据库变更、服务版本、测试环境、联调节奏能够协同起来。

如果这 3 件事都做到了，那么“从改表到联调”就不再是几段割裂的工作，而会变成一条真正可复制的交付流程。
