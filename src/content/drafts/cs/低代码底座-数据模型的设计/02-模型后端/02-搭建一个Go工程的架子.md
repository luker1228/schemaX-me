# 02-搭建一个 Go 工程的架子

## 背景
上一节我们解决的是：**如何调研，如何把一个复杂问题收缩成最小可验证闭环。**

这一节开始，就不再停留在 demo 了。我们来回答另一个更现实的问题：

**当你确认这件事值得做以后，一个 Go 后端工程，第一版架子应该怎么搭？**

很多人一上来就开始写 resolver、写 SQL、写 GraphQL schema，结果三天后项目就开始失控：

今天来教大家如何结合AI coding 来完成这件事情

1. 日志怎么打，怎么把底层库和业务日志统一起来
2. 错误怎么设计，怎么让开发者和 AI 一眼看明白
3. 为什么要做 DDD 分层，边界到底切在哪里
4. interface 层怎么写，repo 层怎么写
5. 可观测性怎么做，才能不是“装了个 Prometheus 就完事”

这节课我就只讲这 5 件事。

这里提一句：这里我在最开始的时候，项目的搭架子初期，我比较推荐直接引用SDD， 我在做比较大的东西的时候， 都会提前使用SDD。这里也不是说强模型弱模型的区分，我单纯的是不希望在这个阶段，让AI发挥创造力。我要的是一个可控的架子。而且类似superpower有brainstorm的功能。这里很容易让AI帮你完成调研。让你去想一些你之前可能忘记想的功能。

## 1. 如何做一个日志门面

### 为什么不要在业务里直接依赖具体日志库

其实我不想解释为什么日志要做一个门面。我可以这么说。作为最流行的语言Java，他保持了这个设计。所以做一个日志的门面模式非常重要。

很多 Go 项目会直接在代码里写：

```go
import "go.uber.org/zap"

func (s *Service) Query(ctx context.Context) error {
    zap.L().Info("query start")
    return nil
}
```

短期看没问题，长期问题很大：

1. 业务代码直接绑死了 `zap`
2. 以后想切 `slog`、想补字段规范、想加 trace_id，很难统一改
3. 第三方库输出格式和业务日志格式不一致
4. 测试时不好替换，不好断言

所以日志最好不要“直接用库”，而是先定义一个**日志门面**。

门面的意义不是炫技，而是：

- 统一字段规范
- 隔离具体实现
- 让业务代码只依赖抽象
- 顺便把 trace、request_id、tenant_id 这类上下文字段灌进去

上面其实是AI总结的优势。这些抛去不看。

这里我只想说一点。做日志门面的最大好处，就是让你的日志规范。看起来是很小的一件事情，但是这个事情决定了你Vibecoding的下限。可以无限提升你Vibecoding后面遇到Harness工程的事情。

> 比如有些人喜欢zaplog这样打日志。 log.Infow
> 有些人喜欢这样。 log.Infof
> 在我这里统统没有，只有一种格式。Infof。即日志里所有的key， 被我强行约束在一个有限集下了。 不会出现， 一会儿requestId， 一会儿reqId， 一会儿requestid 这种问题。

还有一个我非常明确的观点：

**有效日志不怕多，怕的是废话日志。**

什么叫废话日志？就是那种你看完以后，无法帮助你定位任何问题、无法还原任何上下文、无法回答任何工程问题的日志。

比如：

```text
start query
query end
begin handle request
do something failed
```

这种日志最大的特点就是“看起来很忙”，但没有任何信息量。

我不反对多打日志。相反，在复杂系统里，我认为**有效日志就应该足够多**。因为线上排查时，你真正缺的从来不是日志条数，而是关键信息。

所以问题从来不是“日志打多了”，而是：

- 有没有模型 ID
- 有没有请求 ID
- 有没有 SQL / operation / 耗时
- 有没有错误上下文
- 有没有稳定字段名

只要这些东西是有效的，日志多一点不是问题。没有信息量的废话日志，打一百条也没价值。

### 一个足够实用的日志接口

结合我自己在 `modelcraft-backend/pkg/logfacade` 里的落地，我现在更推荐把接口收敛成统一的 `f` 风格：

> 真实代码入口：[`luker1228/modelcraft/tree/master/modelcraft-backend/pkg/logfacade`](https://github.com/luker1228/modelcraft/tree/master/modelcraft-backend/pkg/logfacade)

```go
package logfacade

import "context"

type Logger interface {
    Debugf(ctx context.Context, format string, args ...any)
    Infof(ctx context.Context, format string, args ...any)
    Warnf(ctx context.Context, format string, args ...any)
    Errorf(ctx context.Context, err error, format string, args ...any)
    Fatalf(ctx context.Context, err error, format string, args ...any)

    With(fields ...Field) Logger
    Sync() error
}
```

这里几个点值得注意：

1. 普通日志统一走 `Debugf/Infof/Warnf`
2. 错误日志统一走 `Errorf(ctx, err, format, args...)`
3. `With` 只负责预附加结构化字段
4. `Sync` 负责进程退出前刷盘

为什么一定要带 `context`？

因为后面你真正想要的不是“能打印日志”，而是：

```text
这条日志能自动带上 trace_id、request_id、user_id、model_id
```

没有 `context`，后面这些上下文信息根本接不进去。

### 用 zap 做底层实现，但业务层不要感知

我自己的做法是对外只暴露 `logfacade.Logger`，底层当前用 `zap` 承接：

```go
package logfacade

import (
    "context"
    "fmt"

    "go.uber.org/zap"
)

type ZapLogger struct {
    logger *zap.Logger
}

func (z *ZapLogger) Infof(ctx context.Context, format string, args ...any) {
    if ctx == nil {
        ctx = context.Background()
    }
    msg := fmt.Sprintf(format, args...)
    z.logger.Info(msg, z.extractContextFields(ctx)...)
}

func (z *ZapLogger) Errorf(ctx context.Context, err error, format string, args ...any) {
    if ctx == nil {
        ctx = context.Background()
    }
    msg := fmt.Sprintf(format, args...)
    fields := z.extractContextFields(ctx)
    fields = append(fields, z.extractErrFields(err)...)
    z.logger.Error(msg, fields...)
}
```

核心不是 `zap` 怎么转，而是这里做了两件事：

1. 所有业务日志都统一变成 printf 风格消息
2. 从 `ctx` 里提取 `request_id`，从 `err` 里自动提取 `error + stack`

这样业务代码里就很干净：

```go
repoLogger := logger.With(
    logfacade.String("component", "model-repo"),
    logfacade.String("model_id", model.ID),
)

repoLogger.Infof(ctx, "build graphql schema: field_count=%d", len(model.Fields))
```

如果有错误，也不需要手工 `With(err)`：

```go
repoLogger.Errorf(ctx, err, "build graphql schema failed: field_count=%d", len(model.Fields))
```

`Errorf` 内部直接把 `error` 和 `stack` 结构化打出来。

### 为什么我最后更喜欢这种 `f` 风格

很多人第一次看到这种设计会问：既然强调结构化，为什么消息本身还要走 `Infof`？

因为我在真实项目里最后发现，一个日志接口要想被团队稳定用起来，最重要的是**调用成本低**。

`Infof/ Warnf/ Errorf` 有几个现实好处：

1. 心智负担小，绝大多数同学一眼就会用
2. 文本消息适合快速描述事件
3. 真正需要结构化检索的字段，通过 `With(...)` 预附加就够了
4. 错误场景最容易遗漏的 `error`、`stack`，直接由 `Errorf` 自动处理

所以我的结论不是“不要结构化”，而是：

**消息走 printf，关键字段走 With，错误和堆栈走 Errorf 自动注入。**

再往下一层说，这种设计真正重要的地方，不是“写法统一”，而是**把规范做进硬签名里**。

因为到了 AI 时代，靠约定已经不够了。

你不能指望每个人都记得：

- 错误日志要补堆栈
- 这个字段应该叫 `request_id`
- 这里不要漏掉错误对象

真正靠谱的方式是：**把这些要求收进接口签名和 helper 里，让调用者不容易犯错。**

这才是真正的 harness。

### 日志门面真正解决的，不只是“能打印”

数据模型这种系统，线上最常见的问题从来不是“服务挂了”，而是：

- 某个模型为什么生成出来的 schema 不对
- 某次查询为什么拼出了错误 SQL
- 某条 RLS 策略为什么没命中
- 某个租户为什么查不到数据

这些问题都不是看一个 `panic` 能解决的，而是靠**结构化日志**定位。

所以日志门面真正要统一的是字段语言。比如建议一开始就约定：

- `trace_id`
- `request_id`
- `tenant_id`
- `user_id`
- `model_id`
- `field_name`
- `action`
- `cost_ms`

只要字段语言统一了，后面你接 Loki、ELK、Datadog、ClickHouse 都不难。

## 2. 如何设计错误

### 先定一个硬约束：业务里默认不用标准库 `errors`

这里有一个我自己后来非常坚定的取舍：

**在业务代码里，我默认用 `pkgerrors "github.com/pkg/errors"`，而不是标准库 `errors`。**

原因很简单，不是因为标准库不能用，而是因为它和前面这套日志门面放在一起时，信息密度不够。

你前面已经把日志统一成：

```go
logger.Errorf(ctx, err, "create model failed: model_id=%s", modelID)
```

那接下来一个很现实的问题就是：

**这里的 `err` 能不能天然带出“错误发生点的堆栈”？**

如果你只用标准库 `errors.New(...)` 或 `fmt.Errorf(...)`，很多时候是不带这个能力的。这样 `Errorf` 虽然能打印 `error` 字段，但 `stack` 字段就不完整，最后线上排查价值会打折。

所以我的工程约束就是：

- 业务里创建错误，用 `pkgerrors.New(...)`
- 业务里包裹错误，用 `pkgerrors.Wrap(...)`
- 日志输出错误，一律走 `logger.Errorf(ctx, err, ...)`

这样三件事就串起来了：

1. `pkg/errors` 负责把堆栈挂在错误对象上
2. `logfacade.Errorf` 负责把 `error + stack` 自动结构化输出
3. 排查线上问题时，不需要再手工猜错误是从哪一层冒出来的

比如：

```go
import pkgerrors "github.com/pkg/errors"

func (s *Service) CreateModel(ctx context.Context, input CreateModelInput) error {
    if input.Name == "" {
        return pkgerrors.New("model name is required")
    }

    if err := s.repo.Create(ctx, input); err != nil {
        return pkgerrors.Wrap(err, "Service.CreateModel")
    }

    return nil
}
```

到了接口层或者应用层，只要：

```go
if err != nil {
    logger.Errorf(ctx, err, "create model failed: model_name=%s", input.Name)
    return err
}
```

错误文本、包装链路、堆栈信息就都在了。

这也是为什么我会说：**错误设计不是独立问题，它必须和日志设计一起看。**

这里我还有一个很强的观点：

**错误日志必须打堆栈。**

而且这件事不能靠团队约定，不能靠 code review 提醒，也不能靠“大家记得在出错时多打一行 `With(Stack(err))`”。

这种做法在 AI 时代太脆弱了。

因为无论是人写代码，还是 AI 辅助写代码，只要这个规范依赖“调用者自觉”，它迟早会漏。

所以更稳的方式只有一种：

**把“错误日志必须带堆栈”直接做进 `Errorf` 的接口签名和默认行为里。**

也就是：

- `Errorf(ctx, err, format, args...)`
- `err` 是独立参数，不和格式串混在一起
- logger 内部自动提取 `error + stack`

这样调用者甚至不需要思考“这次要不要打堆栈”，因为签名已经替你做了这个决定。

这不是代码风格问题，而是工程约束问题。

我现在越来越觉得，所谓工程化，很多时候不是写了多少抽象，而是你有没有把那些“必须做对的事情”，通过 API 设计强行收束起来。

### 最差的错误设计：只有一句字符串

很多项目的错误长这样：

```go
return pkgerrors.New("query failed")
```

或者：

```go
return pkgerrors.Wrap(err, "insert failed")
```

这个写法不是完全没用，但对于一个面向开发者的系统来说，信息密度太低了。

开发者真正关心的是：

- 这是什么错误
- 是谁的错
- 能不能重试
- 应该改参数、改数据、还是找平台同学

尤其你这个系统以后要让 AI 来消费，错误更不能只是一段模糊文案。因为 AI 最擅长的不是猜，而是吃结构化信息。

> 这里不是说标准库错误机制“错了”，而是说在一个长期后端工程里，如果你已经明确要把错误和日志堆栈打通，那么 `pkg/errors` 这类带堆栈能力的错误会更顺手。

### 错误要同时服务三类对象

我现在看错误设计，会先问一个很现实的问题：

**这条错误出来以后，谁要消费它？**

答案通常不是一个人，而是三类对象：

1. **开发者**：我要知道怎么改参数、改调用方式、改数据
2. **平台维护者**：我要知道去哪一层排查
3. **程序/AI**：我要能按错误码做稳定分支，而不是靠字符串猜

所以错误最好至少有这些字段：

```go
package errs

type Code string

const (
    CodeInvalidArgument Code = "INVALID_ARGUMENT"
    CodeNotFound        Code = "NOT_FOUND"
    CodeConflict        Code = "CONFLICT"
    CodeForbidden       Code = "FORBIDDEN"
    CodeInternal        Code = "INTERNAL"
)

type Error struct {
    Code      Code
    Message   string
    Op        string
    Err       error
    Fields    map[string]any
    Retryable bool
}
```

这几个字段里，我最看重的是 `Code`、`Op`、`Fields`。因为它们决定了这条错误是不是“可消费”的。

#### `Code`

给调用方稳定语义。

比如重复键冲突，不管底层是 MySQL `1062`，还是将来某个唯一索引校验报的错，对外都尽量收敛成：

```text
CONFLICT
```

#### `Op`

表示“在哪个操作里失败”。

比如：

```text
modelruntime.Repository.InsertRecord
modelruntime.Service.BuildSchema
graphql.Handler.Exec
```

这个字段特别重要，因为线上查错时，`Op` 基本就是第一定位点。

#### `Fields`

补充结构化上下文。

比如：

```go
map[string]any{
    "model_id":  "task",
    "field":     "status",
    "mysql_code": 1062,
}
```

这样无论是日志、HTTP 错误返回、GraphQL error extension，都有东西可挂。

### 一个实用的错误构造方式

```go
package errs

func E(code Code, op string, msg string, err error, fields map[string]any) *Error {
    return &Error{
        Code:    code,
        Message: msg,
        Op:      op,
        Err:     err,
        Fields:  fields,
    }
}

func (e *Error) Error() string {
    return string(e.Code) + ": " + e.Message
}

func (e *Error) Unwrap() error {
    return e.Err
}
```

我更喜欢这种写法的原因，不是“封装得优雅”，而是它把最关键的信息强行收束在一个地方：

- `Code` 给稳定语义
- `Op` 给定位点
- `msg` 给人看
- `err` 保留原始错误链
- `fields` 给结构化上下文

业务里用起来像这样：

```go
if input.Take > 1000 {
    return nil, errs.E(
        errs.CodeInvalidArgument,
        "modelruntime.Service.FindMany",
        "take exceeds max limit",
        nil,
        map[string]any{"take": input.Take, "max_take": 1000},
    )
}
```

这时错误本身就不只是“报错”，而是顺手变成了协议的一部分。

### 错误不是越底层越真实越好，而是越靠近边界越稳定越好

这句话很重要。

很多人做错误设计，会本能地觉得：“底层报什么，我就原样透给上层。”

这在工程上通常是错的。

因为底层错误往往：

- 太技术细节
- 不稳定
- 不适合直接暴露给调用方

比如 MySQL 重复键错误原文很长，字段名也可能是内部索引名。你完全可以在 repo 层识别它，然后转换成领域更稳定的错误：

```go
return errs.E(
    errs.CodeConflict,
    "modelruntime.Repository.InsertRecord",
    "record already exists",
    err,
    map[string]any{"model_id": modelID},
)
```

记住一句话：

**底层错误要保留，边界错误要收敛。**

也就是：

- `Err` 里保留原始错误，方便排查
- `Code` 和 `Message` 对外收敛，方便消费

我自己会把它理解成两层责任：

1. **底层责任**：尽可能保留真实错误、保留包装链、保留堆栈
2. **边界责任**：把错误翻译成稳定、可预期、可编程消费的结果

这样做最大的好处是，系统同时兼顾了两种需求：

- 内部排查时，信息尽量多
- 对外暴露时，语义尽量稳

这和前面日志的思路其实完全一致。我的核心观点始终没变：

**有效信息不怕多，怕的是乱；对外语义不怕收敛，怕的是飘。**

## 3. 为什么要做 DDD 的分层设计

### 不分层，前期写得快，后期一定会反噬

先说一个最常见的“早期顺手写法”：

```go
func (h *Handler) FindMany(w http.ResponseWriter, r *http.Request) {
    modelID := r.URL.Query().Get("modelId")

    rows, err := h.db.QueryContext(r.Context(), "select * from ...")
    if err != nil {
        http.Error(w, err.Error(), 500)
        return
    }

    // 直接拼协议返回
}
```

这种写法的问题不是“不优雅”，而是边界全乱了：

- interface 层知道 SQL 细节
- domain 规则没人承载
- repo 只是 `db.QueryContext` 的同义词
- 将来换 GraphQL、换异步任务入口，逻辑全得抄一遍

而数据模型这种系统，天生就有很强的“多入口、同一套核心规则”的特征：

- HTTP 要调它
- GraphQL 要调它
- 后台任务可能要调它
- 以后 CLI、Webhook、审批流回调都可能要调它

所以核心规则一定不能绑在某一个入口协议上。

### DDD 分层的本质，不是时髦，而是隔离变化

我理解的 DDD 分层，核心不是名词，而是这句话：

**把最容易变化的东西挡在外面，把最核心的规则收在里面。**

对于这个项目来说：

- 最容易变化的是 HTTP、GraphQL、数据库驱动、日志库、监控组件
- 最稳定的核心是模型规则、查询约束、RLS 判定、运行时协议语义

所以理想的依赖方向应该是：

```text
interfaces -> app -> domain
infrastructure -> domain
```

注意：

- `domain` 不应该依赖 `gin`、`gorm`、`mysql driver`
- `domain` 最好也不要关心 GraphQL 的具体对象定义
- `interfaces` 负责把外部协议翻译成内部输入输出
- `infrastructure` 负责把抽象接口落到具体技术实现

### 在这个项目里，各层到底放什么

#### domain 层

这是最高宪法层。

这里放：

- 实体
- 值对象
- 领域服务接口
- 仓储接口
- 领域错误
- 核心规则约束

比如：

```go
package modelruntime

type FindManyInput struct {
    ModelID string
    Take    int
    Skip    int
}

type RecordRepository interface {
    FindMany(ctx context.Context, input FindManyInput) ([]Record, error)
}

type Service interface {
    FindMany(ctx context.Context, input FindManyInput) ([]Record, error)
}
```

#### app 层

负责“组织用例”。

它不一定有很重的业务规则，但它会把 domain 能力编排起来。比如：

- 参数校验
- 调 domain service
- 打日志
- 记录 metrics
- 组织事务边界

也就是说，`app` 更像“应用服务层”。

#### interfaces 层

负责接外部世界。

它只做三件事：

1. 收请求
2. 协议转换
3. 返回响应

最怕的是在这一层偷写业务逻辑。

#### infrastructure 层

负责实现那些“系统总得和外部技术打交道”的部分：

- MySQL
- Redis
- Zap
- OpenTelemetry
- 第三方 SDK

这层的代码通常最脏，但没关系。脏代码不怕，怕的是脏代码往里渗透。

## 3.1 如何做 interface 层

### interface 层的职责：协议适配，而不是业务承载

这一层最容易失控。因为写起来太顺手了。

比如 GraphQL handler 里，很容易顺手就开始：

- 校验参数
- 拼 where 条件
- 查库
- 处理错误
- 打日志

最后 handler 就会越来越胖。

我建议你强制自己守一个规则：

**interface 层最多只做协议翻译，不做核心规则。**

比如 HTTP handler：

```go
func (h *Handler) FindMany(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context()

    input := modelruntime.FindManyInput{
        ModelID: chi.URLParam(r, "modelID"),
    }

    records, err := h.app.FindMany(ctx, input)
    if err != nil {
        writeError(w, err)
        return
    }

    writeJSON(w, http.StatusOK, records)
}
```

GraphQL resolver 也一样：

```go
func (r *QueryResolver) FindMany(ctx context.Context, args FindManyArgs) (*FindManyPayload, error) {
    input := modelruntime.FindManyInput{
        ModelID: args.ModelID,
        Take:    args.Take,
        Skip:    args.Skip,
    }

    items, err := r.app.FindMany(ctx, input)
    if err != nil {
        return nil, toGraphQLError(err)
    }

    return &FindManyPayload{Items: items}, nil
}
```

你看，不管是 HTTP 还是 GraphQL，它们做的事情本质一样：

- 外部协议参数
- 转内部 input
- 调 app service
- 转外部协议响应

真正的业务逻辑都不应该长在这里。

### interface 层最关键的价值：守住边界错误转换

这一层还有一个非常重要的职责：**把内部错误变成外部协议能理解的错误。**

比如：

- HTTP 需要状态码
- GraphQL 需要 `extensions.code`
- gRPC 需要 status code

这时候 interface 层就是最合适的边界。

例如 HTTP：

```go
func writeError(w http.ResponseWriter, err error) {
    e := errs.From(err)

    switch e.Code {
    case errs.CodeInvalidArgument:
        writeJSON(w, http.StatusBadRequest, e)
    case errs.CodeForbidden:
        writeJSON(w, http.StatusForbidden, e)
    case errs.CodeNotFound:
        writeJSON(w, http.StatusNotFound, e)
    default:
        writeJSON(w, http.StatusInternalServerError, e)
    }
}
```

GraphQL 也一样，把 `Code`、`Op`、`Fields` 放到 `extensions` 里即可。

这样好处非常大：

- 领域层不需要知道 HTTP 400/403/404
- repo 层不需要知道 GraphQL error shape
- 不同协议层都能复用同一套内部错误模型

## 3.2 如何做 repo 层

### repo 不是“帮你把 SQL 挪个地方”

很多人以为 repo 就是把 SQL 从 handler 挪到另一个文件。

这理解太浅了。

repo 层真正承载的是：

**持久化语义和领域语义之间的翻译。**

比如领域里说的是：

```text
按模型查询记录
```

但数据库里可能对应的是：

- 动态表名
- 动态字段映射
- where DSL 转 SQL
- orderBy 转 SQL
- 分页限制
- 行级权限条件拼接

repo 层的价值就在于：上层不用看这些细节。

### repo 接口应该定义在 domain，实现在 infrastructure

这点非常关键。

接口定义：

```go
package modelruntime

type Repository interface {
    FindMany(ctx context.Context, input FindManyInput) ([]Record, error)
    Insert(ctx context.Context, input InsertInput) error
}
```

实现放在 `infrastructure/repository`：

```go
type MySQLRepository struct {
    db     *sql.DB
    logger logging.Logger
}

func (r *MySQLRepository) FindMany(ctx context.Context, input modelruntime.FindManyInput) ([]modelruntime.Record, error) {
    query, args, err := buildFindManySQL(input)
    if err != nil {
        return nil, errs.E(
            errs.CodeInternal,
            "modelruntime.Repository.BuildFindManySQL",
            "build sql failed",
            err,
            map[string]any{"model_id": input.ModelID},
        )
    }

    rows, err := r.db.QueryContext(ctx, query, args...)
    if err != nil {
        return nil, mapDBError("modelruntime.Repository.FindMany", err, input.ModelID)
    }
    defer rows.Close()

    return scanRecords(rows)
}
```

### repo 层有两个常见误区

#### 误区 1：repo 直接返回数据库模型

比如直接返回某个 `mysqlRecordDO`。

这样上层马上就会被数据库结构污染。

更稳的做法是：repo 返回 domain 能理解的对象，数据库行结构只在 repo 内部消化。

#### 误区 2：repo 里开始写业务判断

比如：

```go
if user.Role != "admin" {
    // repo 里直接做权限判断
}
```

这通常不对。

repo 只关心“怎么查”“怎么存”“怎么把底层错误翻译出来”。

至于“这个用户该不该查这个模型”，那是更靠近 domain/app 的事情。

当然，像 RLS 这种场景比较特殊，因为它最终一定会落成查询条件。但即便如此，也建议你把：

- 策略判定
- 条件生成
- SQL 拼接

拆成不同步骤，不要一股脑塞进 repo 的一个函数里。

## 4. 如何搭建真正的可观测性

### 真正的可观测性，不是“有日志 + 有监控”就结束

很多团队会说自己有可观测性，实际上只是：

- 有个日志平台
- 暴露了几个 CPU / 内存指标
- Prometheus 能看 QPS

这不叫真正的可观测性。

真正可观测，至少要让你在线上能回答这几个问题：

1. 某个请求慢，到底慢在哪一层
2. 某个租户报错，是只有他报，还是全局都报
3. 某个模型查询失败，是 schema 构建问题、权限问题、还是 SQL 问题
4. 某次发布后，错误率是不是明显升高

如果这些答不上来，你有的只是“监控组件”，不是“可观测性能力”。

### 可观测性的三个支柱：日志、指标、链路

这个概念很多人都听过，但真正落地经常是割裂的。

应该怎么理解？

#### 日志（Logs）

回答的是：

```text
那一刻具体发生了什么
```

适合看细节，适合排查单次问题。

#### 指标（Metrics）

回答的是：

```text
整体趋势有没有异常
```

适合看：

- QPS
- P95 / P99 延迟
- 错误率
- 各类错误码占比

#### 链路（Trace）

回答的是：

```text
一次请求在整条调用链里花了多少时间
```

适合查慢请求和跨组件问题。

### 真正关键的是：三者必须能串起来

如果日志、指标、trace 各搞各的，最后你排查问题还是很痛苦。

所以第一版架子里，建议至少先做到这三件事：

1. 每个请求生成或透传 `trace_id`
2. 日志自动带 `trace_id`
3. 核心指标按 `operation`、`code` 打标签

这样你查问题时，路径就会很清楚：

```text
先看 dashboard 发现 FindMany 错误率升高
-> 再点进 trace 发现主要卡在 repo 查询
-> 再根据 trace_id 去查对应结构化日志
```

这才是一个闭环。

### 第一版最值得上的指标

别一上来埋一百个指标。先埋最有用的：

```text
request_total{operation, code}
request_duration_ms{operation}
db_query_duration_ms{operation}
db_query_total{operation, code}
graphql_resolver_duration_ms{field}
```

为什么这些最有用？

因为它们正好覆盖了两类核心问题：

1. 接口整体是不是健康
2. 数据库是不是瓶颈

### trace 应该埋在哪几层

我建议第一版至少埋在这几个点：

1. HTTP / GraphQL 入口
2. app service 用例入口
3. repo 数据库查询
4. 调外部服务的地方

比如一次 `findMany`：

```text
HTTP /graphql
  -> app.FindMany
    -> repo.BuildSQL
    -> repo.QueryContext
```

这样当某次请求很慢时，你就能直接看出来：

- 是参数解析慢
- 是 schema 构建慢
- 是 SQL 查询慢
- 还是结果扫描慢

### 可观测性的本质是“预埋提问能力”

这是我自己工作里非常深的一个体会。

线上问题出现时，你并不是在“查看系统”，你是在“向系统提问”。

比如你在问：

- 哪个模型最慢
- 哪个错误码最近激增
- 哪个租户经常打到限制
- 哪个 resolver 在拖慢整条链路

如果工程里没有提前把这些字段、指标、span 埋好，那么系统根本回答不了你。

所以所谓可观测性，本质上不是装 SDK，不是接平台，而是：

**在工程设计阶段，就提前想清楚未来你会问系统什么问题。**

## 一个可落地的启动顺序

如果让我重新搭一次第一版模型后端，我会按这个顺序来：

1. 先把 `cmd/server` 跑起来，做健康检查
2. 接一个最小 logger 门面，不要让业务代码直接依赖日志库
3. 先定义统一错误模型，把 `Code`、`Op`、`Fields` 定下来
4. 把 `domain / app / interfaces / infrastructure` 四层目录搭好
5. 先打通一条最小链路：`handler -> app -> repo -> mysql`
6. 在这条链路上把日志、metrics、trace 一次埋通
7. 最后再去扩 GraphQL runtime、RLS、meta、DSL

你会发现，这个顺序和很多人的直觉相反。

很多人会先去做“最有技术含量”的部分，比如动态 schema、SQL builder、RLS 引擎。

但我的经验是：**先把工程骨架搭稳，后面的复杂能力才能长得出来。**

否则你写到一半，就会被这些问题反复打断：

- 错误不统一
- 日志没法查
- 逻辑边界混乱
- 协议层和存储层耦死

最后不是不会写功能，而是项目已经不适合继续写了。

## 这节课的总结

这一节看起来讲的是“工程架子”，但其实讲的是更底层的一件事：

**你到底有没有把一个长期项目，当作长期项目来设计。**

总结成 4 句话：

1. 日志不是打印文本，而是统一上下文和字段语言
2. 错误不是一句字符串，而是开发者协议的一部分
3. DDD 分层不是为了优雅，而是为了隔离变化、保护核心规则
4. 可观测性不是装组件，而是让系统具备回答问题的能力

后面真正进入 runtime、meta、RLS、SQL builder 时，你会越来越强烈地感受到：这些基础骨架不是“前戏”，它们本身就是后端工程的一部分。
