## 先读入口，再拆核心
如果只看 `graphql_app.go`， 很容易误以为它只是一个普通的 GraphQL 执行器。 但它真正做的事情，是把 **模型元数据**、**请求身份**、**权限策略**、**数据库访问** 这四件事情在一次请求里串起来。

可以先把它压缩成下面这条链路：

```text
HTTP 请求
  -> Execute()
  -> 按 org/project/database/model 定位 RuntimeModel
  -> 基于 RuntimeModel 构建或获取 GraphQL Schema
  -> 基于用户身份解析权限并生成 RLS Snapshot
  -> 创建请求级 DB Repo，并包一层 RLSInterceptDB
  -> 把 Repo / Loader / RLSContext 注入 GraphQL Context
  -> graphql.Do()
  -> resolver 从 context 里取 repo 执行真实查询
```

这条链路里面，真正重要的不是 `graphql.Do()`，而是它前面那一长串准备动作。 因为低代码场景下，GraphQL 不是提前写死的，而是运行时按模型动态生成的。

## `graphql_app.go` 到底做了什么
我把这个文件拆成 4 个职责：

### 1. 根据模型定位 schema
`Execute()` 首先把 `orgName/projectSlug/databaseName/name` 组装成一个 `ModelLocator`。 这个对象的意义很简单：它不是在找一个 Go 类型，而是在找一个运行时模型定义。

然后会先调用：

```go
gschema, err := s.GetSchema(ctx, orgName, modelLocator)
```

`GetSchema()` 的逻辑也不复杂：

1. 先尝试从 `GraphqlSchemaManager` 里拿缓存
2. 缓存没有，就从 `modelRepo` 读取模型定义
3. 调用 `NewSchemaFrom(ctx, model)` 动态生成 GraphQL Schema
4. 再把 schema 放回缓存

也就是说，这里的 schema 不是手写 SDL，也不是编译期生成，而是：

```text
RuntimeModel -> GraphQL Schema
```

这就是 runtime 的第一个核心：**模型即接口**。 模型一变，暴露出去的 GraphQL 能力就跟着变。

### 2. 把“谁在访问”翻译成 RLS 上下文
拿到 schema 后，代码没有立刻执行查询，而是先处理权限：

```go
rlsCtx, ctx, err = s.resolveNonAdminRLS(ctx, orgName, projectSlug, modelID)
```

这里的意思非常关键。 在低代码系统里，动态 schema 只是解决了“能不能查”的问题，RLS 才解决“你能查哪些行”的问题。

这段逻辑又可以拆成三步：

1. 从请求上下文中拿到当前用户身份、角色、是否 admin
2. 调用 `permResolver.ResolveFromV2Policy(...)` 算出当前模型上的权限集合
3. 调用 `snapshotBuilder.Build(...)` 把权限集合编译成一次请求可直接消费的 `RLSPolicySnapshot`

这个 `Snapshot` 很重要。 它不是抽象权限描述，而是已经接近数据库执行层的数据结构，比如：

- `SelectFilter`
- `UpdateFilter`
- `DeleteFilter`
- `CreateChecks`
- `UpdateChecks`

也就是说，权限在 GraphQL 入口就被“编译”好了，后面的 resolver 不需要再重复算一遍。

runtime 的第二个核心因此可以总结成一句话：**先把策略编译成快照，再让执行层消费快照。**

### 3. schema 是可复用的，但请求上下文必须是隔离的
接着这段代码特别值得学习：

```go
clientRepo := dml.NewRLSInterceptDB(dml.NewClientDB(clientSqlDB))

reqCtx := modelruntime.WithGraphqlRequestContext(
    ctx, clientRepo, orgName, projectSlug, rlsCtx,
)
```

这里其实在刻意切开两层东西：

- `graphql.Schema` 是类型结构，可以跨请求复用
- `graphqlRequestContext` 是请求级状态，每次请求都必须重新创建

`graphql_request_context.go` 已经把设计目的写得很明确了：

- 当前请求用哪个 `ClientRepo`
- 当前请求有哪些 relation dataloader
- 当前请求属于哪个 org/project
- 当前请求的 `RLSContext` 是什么

这就是 runtime 的第三个核心：**把稳定结构和易变状态分离。**

否则你很容易把 DB 连接、权限信息、loader 这些脏状态塞进 schema 里，最后 schema 根本没法缓存，也没法并发安全复用。

### 4. resolver 不直接管权限，权限下沉到 DB 拦截层
最后一个隐藏得最深，但也是最漂亮的点，是这行：

```go
clientRepo := dml.NewRLSInterceptDB(dml.NewClientDB(clientSqlDB))
```

也就是说，resolver 看到的 repo 已经不是裸 repo 了，而是带 RLS 拦截能力的 repo。

`RLSInterceptDB` 会在真正执行数据库操作前做两类事情：

1. 对读、改、删注入 `USING` 过滤条件
2. 对创建、更新执行 `CHECK` 表达式校验

例如它的读取逻辑大概是这样：

```go
if snap := modelruntime.GetRLSSnapshot(ctx); snap != nil {
    if snap.NoSelectPolicy {
        return nil, errNoPolicy("read")
    }
    injectUSING(ctx, &input.RawFilters, snap.SelectFilter)
}
```

这个设计很重要，因为它意味着：

- resolver 只负责“查什么字段”
- repo 只负责“怎么访问数据库”
- RLS 拦截层负责“这次访问允许到什么程度”

这样权限就不是零散地写在每个 resolver 里，而是统一收口到了数据库访问边界。

runtime 的第四个核心因此是：**把权限控制放在数据访问边界，而不是散落在 GraphQL 字段逻辑里。**

## 所以，runtime 的最小闭环是什么
如果把这套系统继续压缩，我认为它的最小闭环只有 5 个对象：

1. `RuntimeModel`
作用：描述一个模型有哪些字段、关系、能力。

2. `GraphqlSchemaManager`
作用：把 `RuntimeModel` 翻译成 GraphQL Schema。

3. `graphqlRequestContext`
作用：承载一次请求的 repo、loader、org/project、RLSContext。

4. `RLSPolicySnapshot`
作用：把权限策略提前编译成可执行的过滤器和检查器。

5. `RLSInterceptDB`
作用：在真正访问数据库前，把 snapshot 落到 SQL 过滤和写入校验上。

这 5 个东西一旦串起来，动态 GraphQL runtime 就站住了。 剩下的分页、聚合、关系查询、dataloader，本质上都只是能力扩展，而不是底座本身。

## 一个缩小版示例：从模型动态生成 GraphQL
下面我写一个比 `modelcraft` 小很多的示例。 这个示例不追求生产可用，只追求把核心思想讲明白。

### 第一步：先定义运行时模型

```go
package runtime

type Field struct {
    Name string
    Type string
}

type RuntimeModel struct {
    Name   string
    Table  string
    Fields []Field
}
```

这里的 `RuntimeModel` 就是“元数据”。 它不是数据库表本身，也不是 GraphQL 类型本身，而是中间那层统一描述。

### 第二步：定义请求级上下文

```go
package runtime

import "context"

type ClientRepo interface {
    FindMany(ctx context.Context, table string, filters []string) ([]map[string]any, error)
}

type RLSSnapshot struct {
    SelectFilter string
}

type requestContext struct {
    Repo     ClientRepo
    Snapshot *RLSSnapshot
}

type requestContextKey struct{}

func WithRequestContext(ctx context.Context, repo ClientRepo, snap *RLSSnapshot) context.Context {
    return context.WithValue(ctx, requestContextKey{}, &requestContext{
        Repo:     repo,
        Snapshot: snap,
    })
}

func getRequestContext(ctx context.Context) *requestContext {
    v, _ := ctx.Value(requestContextKey{}).(*requestContext)
    return v
}
```

这一步对应 `modelcraft` 里的 `WithGraphqlRequestContext()`。

### 第三步：根据模型动态生成 GraphQL Schema

```go
package runtime

import (
    "fmt"

    "github.com/graphql-go/graphql"
)

func toScalar(t string) *graphql.Scalar {
    switch t {
    case "string":
        return graphql.String
    case "int":
        return graphql.Int
    default:
        return graphql.String
    }
}

func NewSchemaFromModel(model *RuntimeModel) (graphql.Schema, error) {
    fields := graphql.Fields{}
    for _, f := range model.Fields {
        fields[f.Name] = &graphql.Field{Type: toScalar(f.Type)}
    }

    objectType := graphql.NewObject(graphql.ObjectConfig{
        Name:   model.Name,
        Fields: fields,
    })

    query := graphql.NewObject(graphql.ObjectConfig{
        Name: "Query",
        Fields: graphql.Fields{
            "list" + model.Name: &graphql.Field{
                Type: graphql.NewList(objectType),
                Resolve: func(p graphql.ResolveParams) (any, error) {
                    rctx := getRequestContext(p.Context)
                    if rctx == nil {
                        return nil, fmt.Errorf("missing request context")
                    }

                    filters := []string{}
                    if rctx.Snapshot != nil && rctx.Snapshot.SelectFilter != "" {
                        filters = append(filters, rctx.Snapshot.SelectFilter)
                    }

                    return rctx.Repo.FindMany(p.Context, model.Table, filters)
                },
            },
        },
    })

    return graphql.NewSchema(graphql.SchemaConfig{
        Query: query,
    })
}
```

这段代码体现了两个关键点：

1. Schema 是根据 `RuntimeModel` 动态生成的
2. resolver 不自己拼权限，只从 `requestContext` 取 repo 和 snapshot

### 第四步：执行时注入权限和 repo

```go
package runtime

import (
    "context"

    "github.com/graphql-go/graphql"
)

func Execute(ctx context.Context, schema graphql.Schema, repo ClientRepo) *graphql.Result {
    snap := &RLSSnapshot{
        SelectFilter: "owner_id = 'u_123'",
    }

    reqCtx := WithRequestContext(ctx, repo, snap)

    return graphql.Do(graphql.Params{
        Schema:        schema,
        RequestString: "{ listTodo { id title } }",
        Context:       reqCtx,
    })
}
```

这一步对应 `graphql_app.go` 的主入口。 核心不是执行 GraphQL，而是 **在执行前把 repo 和权限上下文准备好**。

## 这套设计为什么适合低代码
因为低代码后端最怕两件事：

1. 模型变了，接口还要手改
2. 权限规则一复杂，所有 resolver 都开始充满 if-else

而 `modelcraft` 这套 runtime 的解法是：

- 用 `RuntimeModel -> GraphQL Schema` 解决接口动态生成
- 用 `RLSContext -> Snapshot -> RLSInterceptDB` 解决权限动态执行

于是你会发现，真正的底座不是 GraphQL 本身， 而是这两个编译动作：

```text
模型元数据 -> GraphQL Schema
权限策略 -> RLS Snapshot
```

这两个编译动作做完，GraphQL runtime 就只是在消费编译结果。

## 我对这个入口文件的结论
`graphql_app.go` 不是一个“查询执行器”，而是一个 runtime 装配器。

它装配了 4 件事：

1. 模型定位
2. schema 生成
3. 请求级权限上下文
4. 带 RLS 的数据库访问

如果你要从 0 开始实现一个复杂的低代码模型后端，我建议第一版不要一上来就做所有 GraphQL 特性。 先把下面这个最小版本做出来：

1. 先定义 `RuntimeModel`
2. 再实现 `RuntimeModel -> GraphQL Schema`
3. 再实现 `RequestContext`
4. 再实现 `RLSSnapshot`
5. 最后给 repo 包一层拦截器

只要这 5 步跑通，整个 runtime 的地基就有了。 后面无论你加的是分页、聚合、关联预加载，还是策略缓存，本质上都只是往这套骨架上继续长肉。
