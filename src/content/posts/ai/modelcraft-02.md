# ModelCraft 02: 从 Schema 到 Runtime，这个项目的主链路怎么串起来

上一篇讲的是问题定义。这一篇开始讲主链路。

我在做 ModelCraft 的过程中，越来越确认一件事：

**一个项目要想长期可维护，必须有一条可以反复复述的主链路。**

如果你每讲一次系统，都要从不同入口开始，那大概率说明系统结构还不够稳定。

## 主链路是什么

对 ModelCraft 来说，这条主链路可以压缩成一句话：

**设计态模型 -> 结构化 Schema -> Runtime 消费 -> 权限执行 -> 对外能力输出**

这五步几乎覆盖了整个项目最核心的路径。

换句话说，我不是先想“怎么生成 GraphQL”，也不是先想“怎么做权限”，而是先把整条链路的因果关系理顺。

## 第一步：设计态模型

一切的起点不是代码，而是模型定义。

这里的关键不是把数据库字段原样照搬出来，而是把它们提升成更稳定的语义层。

例如：

- 一个字段是不是字符串
- 一个字段是不是枚举
- 一个字段是不是关系
- 一个字段是不是可以作为查询条件

如果这些信息只存在数据库细节里，后面所有层都会不停重复判断。那样做，系统会越来越碎。

所以设计态的职责，是把底层事实提纯成可复用的结构描述。

## 第二步：结构化 Schema

当设计态模型被整理出来以后，下一步不是直接跑业务，而是先生成结构化 schema。

为什么这个步骤不能省？

因为 schema 不是中间产物，它是系统的公共契约。

这份契约至少要同时服务三类对象：

1. 前端页面
2. 后端 runtime
3. AI 或其他自动化调用方

如果这三方看到的不是同一份结构，系统就会开始漂移。

最典型的问题是：

- 前端知道一个字段可选，后端却按必填处理
- runtime 支持某个过滤条件，但文档没写
- AI 根据旧描述生成调用，实际接口已经变了

schema 层的作用，就是尽量把这些漂移提前消灭。

## 第三步：Runtime 消费

runtime 不应该是一个“凭感觉解释模型”的黑盒。

它要做的，是严格消费前面生成出来的结构描述，然后把这些描述转成真正可执行的查询、变更和校验流程。

这里我最在意的不是“灵活”，而是“可推导”。

也就是说，当一个字段出现在对外接口里时，我希望自己能顺着链路回答：

- 它从哪里定义来的
- 为什么会出现在 schema 里
- runtime 为什么允许这样查询
- 权限为什么允许或拒绝这次访问

只要这些问题能顺着同一条链路回答，系统就还在控制之内。

## 第四步：权限执行

很多系统的问题，不是没有权限，而是权限挂得太晚。

如果权限只是 runtime 末端的一层补丁，那它很容易变成“每个接口自己记得处理一下”。这种方式初期快，但后面会非常难收拾。

所以我更倾向于把权限看成主链路的一部分，而不是外围附件。

也就是说，权限不是在“拿到数据以后再判断能不能看”，而是尽量提前进入模型执行流程，变成查询条件、执行边界和返回范围的一部分。

这样做虽然设计时更费劲，但系统行为会稳定很多。

## 第五步：对外能力输出

当前面几层稳定下来以后，对外能力才有意义。

不管你最后暴露的是 GraphQL、表单 schema、后台配置能力，还是给 Agent 调用的接口，本质上都只是这条主链路的投影。

这也是我为什么一直强调：

**不要把“接口长什么样”当成系统的第一问题。**

接口是结果，不是起点。

## 为什么我一直强调“主链路”

因为项目一旦复杂起来，人会天然开始局部优化。

看到前端问题，就只修前端。
看到查询问题，就只修查询。
看到权限问题，就只补权限。

这种做法短期很高效，长期会把系统切成很多互相解释不清的小块。

我写这篇文章，其实也是在逼自己持续复述主链路。只有能稳定复述，后面的实现才不容易走偏。
